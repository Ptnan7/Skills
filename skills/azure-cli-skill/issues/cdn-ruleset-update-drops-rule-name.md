# CDN Rule-Set Update Drops `rules[].ruleName`

## Symptom

During a CDN/AFD swagger upgrade, fresh aaz-dev AddSwagger for `/profiles/{}/rulesets/{}` creates `cdn profile rule-set update`, but the local workspace cfg drops `ruleName` from the update `instanceUpdate` input schema.

The broken local cfg shape is:

```text
GET response schema:
rules identifiers: ["ruleName"]
rules item props: actions, conditions, matchProcessingBehavior, order, ruleName, ruleSetName

update instanceUpdate input schema:
rules identifiers: ["ruleName"]
rules item props: actions, conditions, matchProcessingBehavior, order
```

This was reproduced with CDN stable `2025-12-01` in temporary workspace `cdn-ruleset-repro-20260506-125645`.

## Cause

Swagger is not obviously missing the required annotation. `RuleSetProperties.rules` has `x-ms-identifiers: ["ruleName"]`, and `BatchRuleProperties.ruleName` is required.

The issue is that `BatchRuleProperties.ruleName` is create-only. aaz-dev keeps `rules.identifiers = ["ruleName"]`, but filters `ruleName` out of the PUT-based update `instanceUpdate` item props. Treat this as an aaz-dev/AAZ generator bug until the AAZ team fixes create-only nested list identifiers.

## Workaround

Patch only the local `.aaz_dev` workspace cache before Export. Do not hand-edit the `aaz` repo or generated extension output.

Run dry-run first:

```powershell
python .github\skills\azure-cli-skill\scripts\patch_cdn_ruleset_update_cfg.py `
  --workspace <workspace-name> `
  --dry-run
```

Expected output when the workaround is needed:

```text
- added rules[].ruleName to instanceUpdate item props
dry run: no file written
```

Apply the cache patch:

```powershell
python .github\skills\azure-cli-skill\scripts\patch_cdn_ruleset_update_cfg.py `
  --workspace <workspace-name>
```

The helper adds this missing property under `cdn profile rule-set update` `instanceUpdate`:

```json
{
  "type": "string",
  "name": "ruleName",
  "required": true,
  "format": {
    "pattern": "^[a-zA-Z][a-zA-Z0-9]{0,59}$",
    "maxLength": 60,
    "minLength": 1
  },
  "arg": "$resource.properties.rules[].ruleName"
}
```

It also ensures:

```json
"identifiers": ["ruleName"]
```

The helper creates `cfg.json.bak` by default. After the local cache patch, Export the workspace normally so final `aaz` output is still generated by aaz-dev.

After CLI generation, verify the generated extension code contains nested rule-name mapping, for example in `afd rule-set create`:

```python
_element.rule_name = AAZStrArg(
    options=["rule-name"],
    required=True,
)

_elements.set_prop("ruleName", AAZStrType, ".rule_name", typ_kwargs={"flags": {"required": True}})
```
