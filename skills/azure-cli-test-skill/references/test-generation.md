# Test Generation Reference

## Inputs To Inspect

Start from evidence, not guesses:

- Saved swagger diff artifact: `swagger-diffs/<ext>/<old-version>_to_<new-version>.md`. Read this first for modified/new/removed operations, model/property changes, enum changes, breaking changes, and AddSwagger candidates.
- Generated CLI files: AAZ command files under `extension/src/<ext>/azext_*/aaz/latest/` and AAZ command model markdown under `aaz/Commands/...`.
- Generated subcommands: property/subresource command files and command model markdown, especially when generated by `azure-cli-aaz-property-subcommands` from a nested parent resource property rather than a standalone swagger operation.
- Existing tests: closest `test_*_scenarios.py` file and any mixin helper in `scenario_mixin.py`, `afdx_scenario_mixin.py`, or `frontdoor_test_util.py`.
- Existing recordings only as context. Do not regenerate recordings unless the user explicitly approves a live/recording run.

## Diff Coverage Matrix

For swagger/API upgrades, produce and maintain a coverage matrix before editing tests. Every relevant diff item should have an explicit outcome.

| Diff item | CLI surface | Test action | Target file/test | Assertion or reason |
|-----------|-------------|-------------|------------------|---------------------|
| New operation | New command or subcommand | New test or extend setup flow | `test_<feature>_scenarios.py` | Create/use resource and assert returned fields |
| Generated property/subresource subcommand | Nested-property command group such as `add/list/remove` | New test or update existing parent-resource scenario | Nearby scenario file or new subcommand file | Invoke each meaningful verb and assert parent resource property changes |
| Updated operation params/body | Existing create/update/delete command | Update existing test | Nearby scenario file | Send new arg/body shape and assert persistence |
| New response property | Show/list/create/update output | Update existing test | Nearby scenario file | Assert JSON property with `JMESPathCheck` |
| New enum value | Argument or nested object field | Update existing or new focused test | Nearby scenario file | Use enum value and assert service echoes it |
| Removed operation/parameter | Removed command/argument or pinned old API | Skip or negative check only when CLI contract changes | Existing test if needed | Explain unsupported/hidden/pinned surface; avoid brittle service-error tests |
| Breaking change | Changed CLI contract or generated schema | Update affected tests | Existing owner file | Assert new behavior and remove stale expectations |

Group closely related model/property changes when they are exercised through the same command flow, but the matrix must still name the grouped properties. Do not treat generated-code presence alone as coverage; a test should invoke the CLI surface and assert the meaningful returned or persisted value when practical.

For property/subresource subcommands, cover the user-facing verbs rather than the internal parent PUT/PATCH shape. For example, an array subcommand group with `add`, `list`, and `remove` should add one item, assert `list` or parent `show` returns it, remove it, and assert it is gone. Skip a generated verb only when the command is intentionally not registered, duplicates another covered flow, or cannot be made reliable with available service prerequisites.

## Update Existing File Or Add New File

Prefer updating an existing test file when:

- The changed command group already has a scenario file, such as `test_afd_rule_scenarios.py` for `az afd rule` / `az afd rule-set`.
- The swagger diff adds an argument/property to an existing create/update/list/show flow.
- The new behavior can be validated by extending an existing CRUD scenario without making it fragile or too long.

Create a new test file when:

- The swagger diff introduces a new command group with no close existing scenario file.
- The scenario requires a distinct resource setup or should not lengthen a large existing scenario.
- The feature is independent enough to run as its own targeted `azdev test test_<feature>_scenarios` command.

## CDN / AFD Test Patterns

Location: `extension/src/cdn/azext_cdn/tests/latest/`

- Use `CdnScenarioMixin` for `az cdn ...` tests.
- Use `CdnAfdScenarioMixin` for `az afd ...` tests.
- Prefer existing mixin helpers when available.
- Add a mixin helper when the command is reused or matches existing helper style.
- Direct `self.cmd(...)` is acceptable for one-off assertions or a command with no useful helper pattern.
- Use `ResourceGroupPreparer(additional_tags={'owner': 'jingnanxu'})` unless a nearby file uses a stricter preparer.
- Prefer `self.create_random_name(...)` for resource names unless the service requires stable DNS/custom-domain names.
- Use `JMESPathCheck` for response shape and property assertions.

Representative files:

- `test_profile_scenarios.py` for CDN profile behavior.
- `test_endpoint_scenarios.py` for CDN endpoint behavior.
- `test_afd_profile_scenarios.py` for AFD profile behavior.
- `test_afd_endpoint_scenarios.py` for AFD endpoint behavior.
- `test_afd_route_scenarios.py` for AFD route behavior.
- `test_afd_rule_scenarios.py` for AFD rule and rule-set behavior.
- `test_afd_security_policy_scenarios.py` for AFD security policy behavior.

## Front Door Test Patterns

Location: `extension/src/front-door/azext_front_door/tests/latest/`

- Maintained generated/custom test work should target `test_waf_scenarios.py` and `frontdoor_test_util.py`.
- Legacy non-WAF tests are not maintained for swagger-upgrade work unless the user explicitly asks.
- Use existing `WafScenarioMixin` patterns and direct `self.cmd(...)` command sequences.

## Coverage Heuristics

For a swagger/API change, cover the smallest meaningful behavior surface:

- New create/update argument: create or update with the argument, then assert the returned JSON property.
- New list/show response property: create/update a resource that has the property, then assert list/show returns it.
- New enum value: use the new enum in a valid command and assert the service returns it.
- New nested list/object property: verify at least one item with stable identifiers and key nested fields.
- Changed operation request body: validate the generated CLI can send the new body shape, then assert persisted fields.
- New operation or command group: create a focused scenario that proves the operation works end to end, or extend a setup/cleanup flow if the operation is naturally part of an existing CRUD scenario.
- New property/subresource subcommand: exercise the subcommand verbs directly, then assert the parent resource state changed as expected. Do not count a parent create/update test as full subcommand coverage unless it invokes the subcommand surface users will call.
- Removed operation or property: update stale tests and helper code that still depend on it. Add a negative test only when the CLI intentionally exposes a validation/error contract that users depend on.
- Breaking change: prefer updating existing tests to the new contract and documenting why old assertions were removed.
- Error behavior: add an expect-failure or exception assertion only when the swagger change alters validation or a known service error is part of the CLI contract.

Aim to cover all actionable diff items, but avoid broad CRUD rewrites when one focused test or one extension to an existing scenario covers the changed surface. When a diff item is not practical to test, record the skip reason in the coverage matrix.

## Test Run Approval

After edits, propose exact commands but do not run them until approved. Examples:

```powershell
azdev test test_afd_rule_crud --profile latest
azdev test test_waf_policy_managed_rules_sensitivity --profile latest
```

Use azdev discovery keys such as method, class, or test file names. Do not pass raw pytest `::` paths as `<TESTS>` unless you have verified the local azdev version accepts them.

Ask separately for live mode or recording generation:

```powershell
azdev test test_afd_rule_crud --profile latest --live
```
