# CDN/AFD/FrontDoor PowerShell Module Maintenance

## Overview

This skill covers maintaining CDN/AFD/FrontDoor AutoRest-generated Azure PowerShell modules in the `azure-powershell` repo, including `src/Cdn/` and `src/FrontDoor/` when the task is generated-module maintenance. Load the reference file that matches the task.

## Approval Protocol

Ask for explicit approval before editing AutoRest configuration, editing generated or custom source files, deleting superseded test/recording files, running live Azure commands, regenerating recordings, or committing. For ordinary documentation or example edits that the user directly requested, proceed without an extra approval prompt unless the edit deletes content, regenerates content, or changes test recordings.

## PowerShell Test Rules

- Do not use random values, `RandomString`, GUID-generated names, or time-based resource names in PowerShell tests. Use deterministic names that are stable between record and playback.
- Do not manually create, synthesize, or patch recording files. PowerShell test recordings must only be produced by running the test in `-Record` mode.
- Do not add new Pester tests only for object-helper cmdlets or object model types, such as `New-*Object` helper cmdlets, unless the user explicitly asks for them or there is custom behavior beyond simple property assignment.

## Generated File Rules

- Do not manually create, edit, or delete files under `custom/autogen-model-cmdlets`. That directory is generated by `build-module.ps1`; to add or change an object helper, update the model cmdlet configuration/source that drives generation, then run the AutoRest/build flow to regenerate the file.
- After AutoRest/build regenerates `internal/*.ps1`, inspect the corresponding public wrappers under `custom/*.ps1`. If the internal generated cmdlet gained, removed, renamed, or reshaped parameters, update the matching custom wrapper parameters and `process` forwarding logic so the public cmdlet surface stays aligned with the generated internal cmdlet. This applies to hand-maintained custom wrappers such as `New-*`, `Update-*`, `Set-*`, and `Remove-*` scripts, but still excludes `custom/autogen-model-cmdlets` object helper files from manual edits.
- Do not discard generated outputs merely because they look like build noise. Files changed by an approved AutoRest or `build-module.ps1` run, including generated docs, help, examples, test skeletons, manifest exports, solution metadata, `generate-info.json`, and generated/autogen helper files, are part of the generation result and should be reviewed as generated output rather than reverted by default.
- Revert generated output only when it is clearly from the wrong command, wrong branch, wrong module, or an accidental/failed generation run. State that reason explicitly before reverting.
- When generating a new API/module version, run an initial `build-module.ps1` to refresh exports/examples, update the generated `examples/*.md` files, then run `build-module.ps1` again so final docs/help/manifests are generated from the updated examples.

## Environment Setup

One-time initialization (clones `azure-powershell` as `pwsh` and `azure-rest-api-specs` as `swagger` into the workspace):

```powershell
& .github\skills\azure-pwsh-skill\scripts\initialize_pwsh_env.ps1
```

For every new terminal:

```powershell
. .github\skills\azure-pwsh-skill\scripts\use_pwsh_env.ps1
```

This sets `$env:PWSH_REPO_PATH` and `$env:AAZ_SWAGGER_PATH` for scripts and commands below.

## Quick Reference — Swagger Upgrade

1. **Initialize environment** — run `initialize_pwsh_env.ps1` (Copilot)
2. **Activate environment** — run `use_pwsh_env.ps1` in every new terminal
3. **Diff swagger** — load and follow the `azure-edge-swagger-diff` skill. It runs the shared diff script and saves the report under `swagger-diffs/` for later AutoRest, custom wrapper, docs/help, and test planning.
4. **Update README.md** — update commit hash / API version / directives in `.Autorest/README.md` (see [autorest-generation.md](references/autorest-generation.md))
5. **Run autorest** — `autorest` from the `.Autorest/` directory
6. **Review custom code** — check if `custom/` files reference changed models, renamed properties, removed types, or public wrappers for regenerated `internal/*.ps1` cmdlets. When an internal generated cmdlet changes, update the matching `custom/*.ps1` wrapper parameters and `process` forwarding logic if needed. Do not manually edit `custom/autogen-model-cmdlets`; regenerate those object helper files instead. Present findings to the user and wait for approval before editing. See [autorest-generation.md](references/autorest-generation.md) Step 3.
7. **Build module once** — run `pwsh -File ./build-module.ps1` after AutoRest/custom changes so exports and example stubs are available.
8. **Update examples** — run `.github\skills\azure-pwsh-skill\scripts\analyze_module.ps1 -Module <Name>` for new/removed cmdlets + unfilled example placeholders, then update generated `examples/*.md` so examples match the new public surface (see [development.md](references/development.md)). Do not leave examples on previous-version behavior.
9. **Build module again** — rerun `pwsh -File ./build-module.ps1` after example updates so final docs/help/manifests are regenerated from the updated examples.
10. **Test** — ask the user whether to run tests. If yes, run `pwsh -File ./test-module.ps1 -Record` to execute live tests and update recordings (see [testing.md](references/testing.md))
11. **Pre-push CI check** — before pushing or opening/updating a PR, run the local equivalents of CI checks for the touched module: `git diff --check`, targeted playback/record tests, `pwsh -File ./build-module.ps1` if source/custom/help changed, and the same StaticAnalysis categories used by CI Analyze (`example`, `breaking-change`, `signature`, `help`, `ux`, then `check-error`) when artifacts are available. Do not push while any local CI-equivalent check has unresolved errors.
12. **Commit and push** — if tests and pre-push CI checks pass (or any skipped checks are explicitly explained), ask the user whether to commit/push. If yes, stage and commit changes in the `pwsh` repo, then push the branch and create/update the PR.
